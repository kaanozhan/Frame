1 · What & why

Whether a spec's work ships with tests is decided today by whichever agent happens to be planning, at the moment it writes plan.md. Sometimes it does the right thing — audit-q3-generic-any-project lists four test files under ## Files marked New, carries all four into ## Footprint, and folds the authoring into T01/T05/T12. Other specs with equally testable work plan nothing. The mechanism exists; nothing makes the planning run consider it.

Two things follow. Untracked work does not get done: test authoring absent from tasks.md is not counted, not sequenced, not visible to a worker. And unplanned test files break orchestration — ## Footprint feeds the conflict guard, so a worker that invents test/foo.test.js writes outside its declared footprint and trips the conductor's drift check. Planning tests is what makes writing them safe.

The inverse matters just as much: a project with no test setup must not have tests pushed onto it, and a project with only unit tests must not be asked for UI tests it cannot run. Detection gates the whole thing — and, once made, that detection is recorded rather than repeated.

Architectural notes
  • Two signals, two questions — project.commands.test gives the invocation (npm test here, an indirection), the filesystem gives the convention. They are not alternatives to weigh; collapsing them loses one of the two answers.
  • Declared intent, not a guess — detect-project.js:125-127 populates commands.test only for a real, non-placeholder script, so a non-null value means someone declared a way to run tests. It still cannot say whether any test file exists.
  • The probe need not be authoritative. With the user as the fallback and the answer cached, a heuristic that is merely good enough to propose suffices — which is why no runner or harness name has to be hardcoded (C3).
  • Absence is a finding. None detected is written explicitly, so "never looked" and "looked, found nothing" stay distinguishable.
  • Three states, not two. "No tests" and "no way to run tests" are different findings. detect-project.js sets commands.test unconditionally for Rust (:197) and Go (:219) — cargo test and go test ./... work with zero test files — while JS, Python and Ruby require evidence (:125-127, :177-179, :232). So a Go repo with no tests still has runnable infrastructure, and C2 gates on the record rather than the config field: None detected suppresses the question, runner present, nothing written yet does not.
  • Re-verified in one direction only (D4) — when the planned work touches an area the record marks uncovered. That is the only case where staleness changes the outcome, by suppressing tests that have become possible.

Architectural notes
  • Scoped to what runs (D2) — this repo has eight unit-test files and no DOM/UI harness, so a src/renderer/ change is not asked about. Otherwise the plan would carry a test task nobody can execute.
  • Authoring attaches to its code's step, never to a trailing "write the tests" step. A step's work is not done until its tests are.
  • This also removes a duplication risk (D8): with no standalone test task, the coverage matrix in deep-spec-tasks' report pass resolves to a task that already carries the test work, so nothing is created twice.
  • Authoring is planned; verification is not. Writing new test code is work for tasks.md; running existing checks is an implement-time step. Stated explicitly so neither side invents the other's job.
  • "None" is a legitimate answer. A spec with no testable surface — docs, templates, a prompt — plans no tests even where infrastructure exists (S8, and D9 applies it to this very spec).

3 · Decision story

Business decisions

D1 · What should the posture question offer?
chosen
Three options — everything testable / pure logic and data transforms only / none this time
One more option to read; expresses the case that is actually common.
Binary — tests or no tests
Simplest ask, but cannot express a project that tests its pure modules and not its framework-coupled ones.
Why: the middle option is not invented. test/telemetry.test.js states the convention outright — "targets the pure policy module (telemetryEvents.js); the Electron side of telemetry.js is a thin wrapper over it." A binary question could not describe what this repo already does.
D2 · Should the question be scoped to what the project can actually run?
chosen
Yes — the probe distinguishes test kinds
More to establish up front; the gate stays silent where tests are impossible.
One flat "is there test infrastructure?" signal
Simpler and assumption-free, but plans renderer tests in a unit-only project — tasks that cannot be run, or are silently skipped at implementation time.
Why: verified against this repo — eight unit-test files, and none of jsdom, playwright, @testing-library, puppeteer present. Asking about tests for a src/renderer/ change here would produce a plan the project cannot execute.

Technical decisions

D3 · Where do the testing facts live?
chosen
A standing ## Testing section in PROJECT_NOTES.md
Introduces a run-time write outside .frame/; every later run reads instead of scanning.
Detect afresh on every planning run
No new state to keep true, but pays the scan every time and gives the user no place to correct a wrong reading.
Why: converts a per-run cost into a fact recorded once — which is what durable context is for. It also dissolves the hardest constraint in this spec: with the user as fallback and the answer cached, the probe only has to be good enough to propose, so no harness list needs hardcoding and C3 holds without strain.
D4 · When is the record re-verified? follows from D3
chosen
Only when the planned work touches an area the record marks uncovered
A record that goes stale in the other direction is never noticed — accepted, because nothing depends on it.
Re-verify on every planning run
Always correct, and gives back exactly the saving D3 was for.
Why: staleness only changes the outcome in one direction — a record saying "not covered" for an area that has since gained a harness would suppress tests that are now possible. That is precisely the case this rule re-checks.
D5 · Does the run ask before writing the record?
chosen
Write directly; name what was recorded in the closing message
A shared meta file changes without an explicit prompt each time.
Confirm before each write
Full control over shared state, at the cost of an extra approval step on every planning run.
Why: REFERENCE.md already grants it — "You can add without asking the user (for important decisions)" — and a confirmation on every run is the kind of noise that trains people to click through.
D6 · Which part of PROJECT_NOTES.md? follows from D3
chosen
The standing topical area, beside ## Tech Stack
None — it matches how the file is already organised.
A dated entry under ## Session Notes
Follows the documented note-taking convention, but a cache that is rewritten does not belong in an append-only log.
Why: decided silently — the file already draws this line. Lines 1–396 are standing reference sections updated in place; ## Session Notes from line 397 is the dated append-only log. The record is a fact about the project, like Tech Stack, not an event.
D7 · Is the posture also written to status.json?
chosen
No — plan.md carries it
A future implement flow would have to read the plan rather than a field.
Add an implement_mode-style field now
Ready for a consumer that does not exist yet.
Why: decided silently — nothing reads such a field today. implement-modes is a separate, unimplemented spec; adding state with no reader is speculative.
D8 · Is a dedup rule needed against deep-spec-tasks?
chosen
No — the sequencing rule already prevents it
Depends on that rule holding; if a standalone test task ever appears, the risk returns.
Add an explicit dedup rule to the report pass
Belt and braces, but a rule guarding a case the design does not produce.
Why: decided silently — the spec flagged that a test task could arrive twice, from ## Sequencing and from the report's coverage matrix. Because authoring is folded into the step producing the code, no standalone test task exists to duplicate: the coverage ID resolves to a task that already carries the test work.
D9 · Does this spec's own work ship with tests?
chosen
No — "none this time"
The change ships unverified by machine; it is a prompt, exercised by running it.
Add a test for the template
Would assert the presence of prose in a markdown file — a test of the diff, not of behavior.
Why: decided silently — applying D1 to this spec honestly gives "none this time". Recorded rather than left unsaid, because planning a test for a markdown template is exactly the failure mode D2 exists to prevent, and the spec should be seen obeying its own rule.

Deliberately deferred: a machine-readable form of the record (it stays prose a human edits); teaching spec.new / spec.implement to read it; letting the implement flow consume the posture decision (belongs with implement-modes); setting up test infrastructure where none exists — Frame detects, it does not install.

  1. Iteration 1 — coverage audit: D3–D5, the PROJECT_NOTES.md record, sit under no Goal in spec.md. Rather than fold them in silently, added an explicit "this plan extends the spec" subsection to Architecture naming what was added and when it was decided.

Iteration 2 — constraint check: C1 says spec.plan.md is the only file this spec changes, which stays true of the repo diff, but the command now writes to PROJECT_NOTES.md at run time — a side effect no other spec command has. Separated the two readings in the scope note instead of claiming C1 is untouched.
  3. Iteration 3 — task-derivability: the probe was a single step bundling read, detect, ask and record, which would have produced one oversized task. Split into four; the remaining three steps were already independently shippable.
  4. Iteration 4 — format and coverage audit: five sections in order, every G1–G5 / C1–C7 / S1–S8 ID owned by a named section, zero TBDs, every referenced path and symbol verified in code, tasks derivable from Sequencing alone. Pass — converged.
  5. Iteration 5 — post-review, prompted by the question "won't the record detect this regardless of language?". It will — the filesystem side of detection always runs, so the record is language-independent by construction and the Rust/Go asymmetry in commands.test cannot mislead it. The review did surface a real gap in the opposite direction from the one first suspected: a Go repo with zero tests has runnable infrastructure, so treating "no tests" as "no infrastructure" would wrongly suppress the question. Split the record into three states, moved C2's gate onto the record rather than the config field, and added Sequencing step 3. No language-specific rule was needed — the record was already expressive enough.
