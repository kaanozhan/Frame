# Spec Knowledge Layer — specs as durable agent + human memory

> Captured from the 2026-07-20 design session. A standalone layer — deliberately
> **not** coupled to `audit-q3-deterministic-graph-hints` (separate spec, separate
> script, separate data; they only happen to use the same Claude Code hook
> facility). Discussed at length, not yet planned.

## Problem

Frame's spec archive already records the most expensive knowledge in the
project: **what** was done, **why** that path was chosen (including rejected
alternatives), **how** it was executed, and **what the outcome was**
(spec.md → plan.md → tasks.md → outcome.md). But nothing delivers that
knowledge to an agent at the moment it matters. Today an agent editing
`tasksManager.js` has no idea the perf spec added a parse-once cache and a
self-write guard there, or that cross-platform deliberately rejected keytar —
so it re-does archaeology from raw code every session, and can confidently
re-propose things that were already tried and rejected.

An AGENTS.md instruction ("scan the specs first") is not a mechanism — the
same lesson `audit-q3-deterministic-graph-hints` already recorded: whether an
agent honors advisory text depends on it reading and remembering it, every
session, every time. The user's requirement is explicit: **this must always
work.**

## Vision

Two lookups, both guaranteed:

1. **Topic lookup** — given a subject (a user prompt, a new spec idea), find
   the specs that touched it and understand the context.
2. **File-history lookup** — given a file about to be changed, surface every
   spec that touched it: what was done, why, how, and the result.

The compact layer is an *entry point*, never a substitute: on a hit, the
agent follows the chain down — `digest → spec.md → plan.md → tasks.md →
outcome.md` — and does the real understanding on the raw artifacts.

## Architecture

### 1. Source artifacts stay as they are

spec.md / plan.md / tasks.md / outcome.md remain the human-facing,
deep-reading layer. Nothing in this spec changes their format except the
additive authoring fields below.

### 2. `digest.md` — per-spec distillation (new artifact)

Written by the agent as a step of `spec.done`: 10–15 lines — what was done,
why this path (rejected alternatives included), the result, rules
established — plus a structured `keywords:` line. The digest is the
"written for agents" version; outcome.md stays the depth behind it.

### 3. Spec authoring fields (template change, birth-time enrichment)

Specs are agent-authored into a known structure, so relationships are
**declared at birth, not inferred later**. `spec.new` gains:

- a catalog-evaluation step: Frame embeds the full spec catalog (one compact
  line per spec) into the dispatched command file; the agent — the strongest
  semantic matcher available — decides relatedness *while authoring*, and
  lets prior decisions shape the new spec (conflicting decisions →
  Constraints, overlap → Out of scope, open forks → Open Questions);
- structured fields the index reads verbatim:
  `keywords:`, `related:`, `supersedes:`.

At the current scale (~28 specs) embedding the whole catalog is trivial —
recall is 100% and precision lives in the model. Lexical scoring
(normalization, Turkish/English folding, synonym map, rarity weighting,
top-N cap) exists as the scaling path and for backfill, not the v1 hot path.

### 4. Derived index — `.frame/index/spec-index.json`

Generated, never hand-edited, always reproducible. Two views:

- **`topics`** — per-spec catalog records built from declared `keywords:`,
  titles, template-guaranteed headings, and footprint file-name tokens.
- **`files`** — file path → **list** of entries (never a single record):
  `{slug, task, what/why one-liner, result, date, deep-read path, flags}`.
  Sources: plan `## Footprint` (intent), outcome `Files touched:` (actual),
  commit `(slug)` tags (enrichment only — **primary sources are Footprint +
  outcome**, so squash-merges can't hollow the index out).

Regenerated by Frame on spec phase transitions (exact trigger is a plan
question). Path keys are posix-normalized (`/`) on every OS.

### 5. Query CLI — `scripts/spec-context.js`

Sibling of `find-module.js`, two modes:

```bash
node scripts/spec-context.js <keyword>            # topic → specs + digest lines
node scripts/spec-context.js --file <path>        # file → chronological history
```

Output is deliberately compact: a few lines per hit + a deep-read pointer
(`.frame/specs/<slug>/…`). The agent pulls depth itself; the index never
manages the agent's context budget for it.

### 6. Delivery — two deterministic channels + one fallback

**Hooks (unpredictable moments — interactive/worker implementation).**
Claude Code PreToolUse/UserPromptSubmit; neither relies on the agent
remembering anything.

- **Edit/Write only** — deliberately *not* Grep/Glob/Read: exploration
  touches dozens of files, edits touch a handful; hooking reads would bloat
  memory with history of files that turn out irrelevant. The injection point
  is the moment of intent, not the moment of exploration.
- **Session dedup** — one injection per file per session (state keyed by the
  hook's session id; stale state files cleaned up).
- **Budget with overflow-to-pointer** — 1–2 entries: full compact records;
  3+ entries: one line per spec + pointer. **All specs are always captured**
  — overflow reduces depth per entry, never drops an entry.
- **UserPromptSubmit** — topic scan once per user message; stopwords,
  match threshold, max-N cap so generic words ("fix", "panel") can't flood.
- Both a **full-content mode** and a **signal-only mode** ("this file has
  spec history — run spec-context.js before editing") are implemented;
  the default is decided by dogfooding (plan question).

**Template/dispatch injection (predictable moments).** Frame composes these
prompts in code, so a step here is as guaranteed as a hook:

- `spec.new` — catalog evaluation + authoring fields (see §3).
- `spec.plan` — mandatory evidence-pass step: `spec-context.js --file` for
  every candidate Footprint file, so decisions are made against recorded
  history; plus in-flight footprint intersection ("this file is in the
  footprint of spec X, currently implementing") — interactive sessions get
  the collision awareness today only the conductor has.
- Worker prompts — `buildWorkerPrompt` preloads the spec's footprint file
  histories; workers start armed before their first tool call.
- `spec.done` — digest authoring + index regeneration.

**AGENTS.md (fallback).** Advisory lines next to find-module for non-Claude
CLIs, where hooks don't exist. The team is effectively 100% Claude Code, so
the "always works" goal holds in practice.

### 7. User-enlightenment protocol

The injected text carries not just data but a **behavior instruction**:
*"if this history affects your work, relay it to the user in 1–2 sentences
before changing the file; if not, stay silent."* Recency makes this far more
reliable than a session-start rule — the instruction arrives as the newest
context at exactly the right moment. Purpose is dual: UX (the agent visibly
works history-aware) and human memory (the developer is reminded of a
6-month-old decision at the exact moment it matters). Once per file, never a
history lecture.

### 8. Frame UI surface (companion, separate task)

Since the index sits on disk, Frame can render a "Spec history" timeline for
the selected file (file tree / editor), linking into outcome.md — the same
memory for the human even when no agent is running. Core layer works without
it; v1 inclusion is a plan question.

## Correctness requirements

- **Full history, chronological.** The file view returns *every* spec that
  touched the file, oldest→newest, newest marked as current truth; older
  entries explain how the code evolved.
- **Supersede awareness.** New specs declare `supersedes:` at birth; for
  legacy entries the index computes "later specs also touched this file"
  flags from git. The dangerous case — reading March's "why" without seeing
  July reversed it — must be impossible to miss.
- **Stale flag.** File changed by non-spec commits after the spec closed →
  entry flagged; wrong-but-confident context is worse than none. This is the
  layer's seatbelt, present from v1.
- **Rename/move/delete tracking.** Index build follows git renames
  (`--follow`); deleted files keep their records with a "logic moved to X"
  link. (The repo's own history — escapeHtml → htmlUtils.js, planned
  escapePathForShell → shellEscape.js — is exactly this case.)
- **Phase filter (what enters the index):** `done` + outcome → full records;
  `implementing`/`planned` → in-flight warnings only; `specified` → topic
  catalog only (keywords exist from birth), no file view; test/superseded
  specs → excluded.

## Constraints (hard lines from the discussion)

- **A separate layer.** Independent of `audit-q3-deterministic-graph-hints`
  in every artifact; shared harness feature only. Either can live or die
  without the other.
- **Never block, never break.** Hooks return additive context only; any
  failure (missing/corrupt index, bad stdin) → exit 0, silent. A hook error
  must never surface as a tool error.
- **Speed budget.** Hook scripts read the pre-built index only — no live
  grep over `.frame/specs/`, no Frame IPC. Target ~50 ms including node
  startup. The hook must work with the Frame app closed (agents run outside
  Frame too); Frame's only runtime role is keeping the index fresh.
- **Dependency-free scripts**, shipped to user projects like their siblings
  (`find-module.js` pattern).
- **Index is generated.** Never hand-edited; orchestration workers never
  regenerate it (extends the "never touch meta files" rule).
- **Context discipline.** No injection on read/exploration; dedup + budget
  as specified; the topic scan is bounded per prompt.

## Prerequisite & backfill

- **Inventory hygiene first:** close/mark test-orch-1..4, superseded
  `structure-non-standard-layouts`, and stale `implementing` phases — the
  index is only as good as the archive it derives from, and this layer is
  what finally makes spec hygiene load-bearing.
- **One-time backfill:** generate digests + `keywords:`/relationship fields
  for existing done specs (lexical scoring assists here).

## In scope

- `digest.md` authoring step in `spec.done` template.
- `spec.new` template: catalog-evaluation step + `keywords:/related:/supersedes:` fields.
- `spec.plan` template: footprint history evidence step + in-flight intersection.
- Worker prompt preloading in the dispatch path.
- Index builder (`.frame/index/spec-index.json`) + regeneration trigger.
- `scripts/spec-context.js` (topic + `--file` modes).
- Hook script(s) + merge-safe settings install (Edit/Write PreToolUse,
  UserPromptSubmit), gated on `ai_tool: claude-code`.
- User-enlightenment instruction in the injection template.
- Session-dedup state + cleanup.
- Inventory hygiene pass + backfill.
- Eval: injected vs. non-injected comparison — "context is not
  comprehension"; the hypothesis that injection improves agent behavior is
  tested with data, not assumed (mirrors the graph-hints eval philosophy).

## Out of scope

- Hooking Grep/Glob/Read (explicitly rejected — context bloat).
- Embeddings / vector search (API + infra dependency, no gain at this
  scale; the catalog + model semantics do the job; revisit at hundreds of
  specs).
- Any coupling with the code graph / graph-hints artifacts.
- Blocking or gating any tool call.
- Indexing non-spec work beyond commit-tag enrichment (accepted boundary).

## Success criteria

- Editing a file with spec history in a fresh session visibly yields the
  chronological history (all specs, newest marked) with zero reliance on
  AGENTS.md — and the agent relays the relevant part to the user.
- A new spec authored via `spec.new` arrives with `keywords:`/`related:`
  filled and its content shaped by prior decisions; the resulting
  `spec.plan` shows footprint-file history in its evidence pass.
- A file renamed since its spec closed still resolves its history.
- 3+ overlapping specs on one file: every one is present in the injection
  (overflow drops depth, never entries).
- A project with no index behaves exactly as today (silent, zero errors);
  hook overhead stays within budget.
- Eval shows injected sessions consult and apply spec history more than
  non-injected ones (directional evidence suffices).

## Open questions for /spec.plan

- Index regeneration trigger: specManager phase transitions vs. watcher vs.
  explicit `npm run spec-index` (or a combination).
- Default injection mode: full-content vs. signal-only (decide via
  dogfooding; both are cheap to implement).
- Hook install location for user projects: `.claude/settings.json` vs.
  `settings.local.json`.
- Digest format: front-matter vs. plain sections; exact field list.
- Frame UI file-history panel: v1 or follow-on.
- Shape of the in-flight footprint warning (hook-time vs. plan-time only).
- Does the existing eval harness support hook-enabled runs (same question
  graph-hints has open — answer once, share nothing else).
