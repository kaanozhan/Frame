1 · What & why

/spec.implement ships exactly one flow: take the next pending task, ship it, write an outcome entry, stop. The user turns the loop by dispatching again. That is a reasonable default and the only option.

Two gaps follow. There is no autonomous path — someone who trusts the plan and wants the whole spec built in one run must dispatch N times and review between each. And there is no way to bring your own flow: teams have conventions (test-first, no-commit-until-review, verification steps specific to their stack), and today the only lever is replacing spec.implement.md wholesale via the template override, which also discards everything Frame's default flow does for task and phase accounting.

This adds a mode picker with three choices, a rich mode that runs the whole range with tests, atomic commits and an HTML report, and a described-flow mode that layers a user's own procedure on top of a core it cannot drop.

Architectural notes
  • Asked before the work, not after (D1). A task executed before the answer obeys no mode — uncommitted, absent from the report — and a rich run would then need retroactive repair.
  • Asked every time, answered in one keypress (D1, revised on the user's direction). A saved default leads the list marked (default) rather than silencing the question — which makes switching modes mid-spec free: run the first tasks step by step, hand the rest to the autonomous mode. status.json now records the last choice, not a never-ask-again resolution.
  • The upgrade seam is one re-dispatch (D10). Frame launches on a hint and cannot know the answer; picking more autonomy than the session has means record-and-relaunch, while picking less proceeds — step-by-step's control point is the agent's own per-task question, not the permission layer.
  • The picker sells the payoff, not the label. The autonomous option must state that it produces an HTML report openable as each task completes, with that task's real diff, a what-changed / why-changed summary, and its test result.
  • Additive state (C4): validateSpecStatus (specManager.js:91) checks required fields only and every writeStatus spreads the existing object, so implement_mode needs no schema change.
  • No assumption about how to ask (G7): use the CLI's structured-question tool if it has one, otherwise print numbered options and stop — never self-select and continue.

Architectural notes
  • The amend exists because a hash cannot precede its own commit. The entry is written with an empty hash, the commit is made, the hash is read back, the report is regenerated and folded in with --amend --no-edit. Safe precisely because nothing is pushed.
  • Diffs are never transcribed — the generator pulls them from git by hash. That is the one place a hallucination would silently corrupt the artifact, and the whole reason the report is generated rather than written by the agent.
  • No confirmation between tasks (D2). "Ask almost nothing" is the mode's reason to exist; a prompt per task turns it back into step-by-step.
  • Verification comes from the project (C3): .frame/config.json project.commands, with no runner or stack baked into the shipped template.
  • The report excludes its own bookkeeping — per-commit diffs drop .frame/ and the report files, so what is shown is implementation.

Architectural notes
  • A rule is not an interactive approval. Verified in the docs: an interactive Edit/Write approval lasts only for the session and is never written to disk; only Bash approvals persist. So "the user grants it once in the CLI" cannot work for a mode whose main activity is editing files — a pre-existing rule is the only mechanism that does.
  • The denylist carries the safety load, not the allowlist. dontAsk was considered and reversed (D4): it silently denies everything unlisted with no way to ask, so any unanticipated harmless command stalls the run. In auto, ask-when-it-matters replaces deny-and-stall — repeated classifier blocks pause the run and put a real question to the user.
  • C2 survives — the file lives under .frame/, and --settings takes an arbitrary path and loads additional settings, so .claude/ is never written and the user's own rules stay in force.
  • "Never push" stops being prose (D9, C7). Bash(git push *) in the denylist, and deny is evaluated first at every scope, so the constraint becomes mechanically impossible rather than requested.
  • Frame supplies the runtime (D6). ELECTRON_RUN_AS_NODE=1 on Frame's own binary runs as Node 18.18.2, injected into the terminal the way envForLane already injects orchestration variables. The generator must therefore target Node 18, though the repo's CI runs Node 20.

Business decisions

D1 · When is the mode chosen? revised on review
chosen
At every dispatch, before any work — default listed first, marked, one keypress
One question per dispatch; in step-by-step that means one per task.
Once per spec, persisted, never re-asked
Zero repeat questions — and switching modes mid-spec means digging into state instead of just answering differently.
Why: first decided as ask-once; revised on the user's direction. Always-asking makes mode-switching free — run the first tasks step by step, hand the rest to the autonomous mode — and with the default leading the list the cost is a single Enter. What survives from the original decision is the before any work half: a task executed before the answer obeys no mode, and a rich run would need retroactive repair. status.json keeps recording the choice, but as the next dispatch's launch hint rather than a silencer.
D2 · Does the rich mode confirm between tasks?
chosen
No — commit and continue
A wrong turn runs further before the user sees it.
One short confirmation per task
Keeps the user in control, and makes the mode indistinguishable from step-by-step.
Why: "ask almost nothing" is the mode's reason to exist. Nothing is pushed, so an unwanted commit is cheap to undo — and the report plus outcome.md mean the user can review the whole run afterwards rather than gate it step by step.

Technical decisions

chosen
Pure logic and data transforms only
The git-facing and filesystem paths go uncovered.
Everything testable, with a temporary repo fixture
Broader assurance; starts a fixture convention the project does not currently have.
Why: the first run of the new posture gate. The testing record's Covered line states the convention — test the pure module, skip the Electron-coupled wrapper — so the option matching it led. D4 · How does the autonomous mode act without a prompt per step? revised on review
chosen
auto mode plus a denylist Frame owns under .frame/
Trusts the background classifier for unlisted operations.
dontAsk plus an exhaustive allowlist
Strictest on paper — and any unanticipated harmless command silently stalls the run, with no way to ask.
Why: first decided as dontAsk, reversed when the user asked the right question: "if it's not on the list, it just denies? shouldn't it ask when asking is what's needed?" Exactly so. An exhaustive allowlist cannot be written in advance — mkdir, a rename, a formatter would each stall a dontAsk run with no recovery. In auto mode, deny rules still resolve first in every mode (the push denial keeps its teeth — needed, since the classifier otherwise permits pushing to the working repo by default), unlisted harmless operations are approved by the classifier, dangerous ones are blocked, and after 3 consecutive or 20 total blocks the session pauses and prompts the user — ask-when-it-matters instead of deny-and-stall. --settings still takes an arbitrary path and loads additional settings, so the file lives under .frame/ and C2 holds.
D5 · Where does a described flow live?
chosen
Its own file, referenced from config
Two files instead of one, and a reference to resolve.
Inline in config.json
Single source of truth, stored as an escaped multi-paragraph JSON string.
Why: a description runs to paragraphs and config files are read whole. Embedding prose there makes both the config unreadable and the flow painful to edit — the user would be fighting JSON escaping to change how their own work runs.
D6 · Where does the Node runtime come from? follows from the report generator
chosen
Frame supplies its own, injected into the terminal
The generator must target Node 18, not the repo's CI version.
Require node on the user's PATH, skip the report when absent
Nothing to wire up, and the headline feature silently disappears for anyone without a global Node.
Why: verified — ELECTRON_RUN_AS_NODE=1 on Frame's binary runs as Node 18.18.2, so the runtime is already present and depending on the user's PATH is self-inflicted. Frame is the tool; supplying what it needs is its job. envForLane (orchestrationManager.js:87) and ptyManager.js:217 are the existing path for handing a terminal an environment.

chosen

D9 · "Never push" — prose or mechanism?
chosen
A deny rule
None — the allowlist exists anyway.
An instruction in the prompt
How it works today, and it depends on the agent obeying.
Why: decided silently once D4 landed. Deny is evaluated first and cannot be overridden at a lower scope, so C7 becomes impossible rather than requested. History rewrites beyond the mode's own --amend are denied on the same grounds.
D10 · Frame launches before the picker answers — how do the flags get set? follows from D1 + D4
chosen
Launch on a hint; an autonomy upgrade costs one re-dispatch
Rarely, the user answers the picker and then presses Run once more.
Always launch with autonomous flags, just in case
No seam ever — and every step-by-step and planning session runs with broader permissions than it needs.
Why: the permission mode is a launch flag and cannot be changed from inside a running session, while the picker's answer only exists after launch. The hint — this spec's last choice, else the project default — is right in the common case; when it is wrong in the upgrade direction the agent records the choice and requests one re-dispatch, because an autonomous run without its flags would prompt on every edit, the exact thing the mode exists to avoid. Wrong in the downgrade direction is harmless: step-by-step's control point is the agent's own per-task question, not the permission layer.

The autonomous mode makes a wrong turn and runs several tasks past itAccepted as the cost of D2. Nothing is pushed, every task is its own commit, and outcome.md plus the report make the run reviewable afterwards — so recovery is git reset, not archaeology.
The verification command in project.commands is wrong or absentResolved from the project rather than guessed. When absent, the mode proceeds without a check rather than inventing one — a wrong command would fail every task and stall the run.

Two specs run autonomously in parallel and both write the permission fileThe file is derived from the mode rather than the spec, so concurrent writes agree. Noted rather than guarded.
