Implement modes v2 — the mode is chosen before the session, never inside it

2026-07-22 · phase specified → planned · canonical artifact: .frame/specs/implement-modes-v2/plan.md · produced by /spec.plan

13 decisions taken3 loop iterations2 drifted claims caughtsupersedes autonomous-permission-lifecycle

1 · What & why

v1's mode picker lives inside the CLI session, but the autonomous mode's permission flags can only travel on the launch line — so choosing autonomous after launch can only end in a "please re-dispatch" loop. The rejected fix (merging Frame's permission rules into the user's .claude/settings.local.json) put ephemeral state in a user-owned, repo-scoped file and still didn't silence prompts. Meanwhile the implement button's copy and turn-scoped lock hardcode step-by-step semantics for every mode, and nothing exists between "approve every task" and "full autonomy".

v2 moves the choice before the session: one unified modal (mode + destination) writes implement_mode to status.json first, then stages, then dispatches — flags derive from the recorded choice, never from a guess. A new guided mode fills the ladder gap (continuous loop, the CLI's own permission prompts pace it). Autonomous becomes launch-path only, with a new .frame/bin/implement-launch.js helper as the single source of the flagged launch line for terminal users. Nothing ever touches .claude/settings.local.json.

Architectural notes
  • Re-dispatch is unreachable from the UI: the mode is recorded before staging, so getImplementLaunchFlags (which already reads status.implement_mode first) needs zero change — the flip alone makes the flags correct on the first launch.
  • Mode × destination coupling: autonomous enables "Continue in <Frame>" only when the lane's own launch carried the flags (lane.launchedAutonomous — new tracking, see D8); otherwise disabled with the reason shown and destination forced to a new Frame. The old lane is unassigned, not closed.
  • Flags refused mid-dispatch: the existing bare-retry now states a fallback to guided (D3) — notice + prompt note both say so; the recorded mode stays autonomous.
  • _askContinueOrNew survives for spec.new/plan/tasks — absorption applies only where a mode exists.

D1 · What is the middle mode's final name (UI copy + persisted value)?
chosen
guided
Matches the spec's own vocabulary ("Start Guided Run"); reads as "the CLI's prompts guide the pace".

Emphasizes the user as permission gate, but reads heavier in button copy and diverges from the spec text.
Why: asked at the gate; the string is both UI copy and the implement_mode value, so it had to settle before any code. User chose guided.

chosen
Fourth modal slot + CLI-conversational body + skill save
Modal lists it; the description itself stays conversational; a finished described run offers to become a project skill the CLI can detect and re-run.

Supersedes v1's .frame/implement-flow.md + flowFile for claude-code. The skill is written by the CLI agent at run time — no Frame code path writes under .claude/.

Why: asked at the gate. The recorded mode stays autonomous — it is the user's actual choice and the next flagged launch should honor it.
