1 · What & why

/spec.plan now ships two artifacts: plan.md, the strict five-section contract, and plan-report.html, the reasoning the strict format cannot carry — risk matrix, coverage matrix, decision story, verified claims, convergence log. /spec.tasks predates the report. It reads only spec.md and plan.md, so everything the planning run learned and recorded outside plan.md is invisible at the moment tasks are generated. Risks with no owning task become nobody's work; a success criterion whose owning plan section produced no task is a silent hole. The task list is the last artifact before code and the only one an orchestration worker executes, so a gap here is a gap that ships.

The fix is a second pass, not a bigger first pass. The plan pass derives a complete list from plan.md alone; the report pass then audits that list against the report and closes what it finds. Keeping them separate is what preserves plan.md as the spine — a single pass reading both would let the report's narrative prose dilute task granularity, and would blur which document is authoritative when they disagree.

Architectural notes
  • The first pass is self-sufficient: it must produce a shippable list on its own. The report may add and correct, never weaken — that is what keeps plan.md the spine.
  • Authority is asymmetric: the report denounces gaps; it is not a second source of truth. Every conflict resolves in favour of plan.md.
  • The write stage emits nothing new: heading plus - T<n> · … bullets. Reconciliation leaves no trace in the file (silent reconcile), so downstream parsing is unchanged.
  • Sections are located semantically (D2) and the report is read whole (D3) — no binding to "4 · Risks & edges" as a literal string.

Architectural notes
  • Nothing is ever renumbered (D8) — for any task, in any status. This is the entire defence: specManager.js:441-443 overwrites a matched task's title in place while preserving its status, so a shifted ID would transplant a completion flag onto work never done. That path is reachable only by renumbering, so declining to renumber makes it unreachable, not merely unlikely.
  • Ordering and identity are separate concerns. File order carries implementation order; the ID is only an identity. Ascending order is deliberately sacrificed — a non-ascending list is correct output, not a defect.
  • Status is not consulted (D6). With reassignment off the table there is no "safe to move" category to detect, so tasks.md alone suffices and tasks.json is never read.
  • Numbers outgrow the count. Allocation is one-way, so a 9-task list may end at T14. The 5–12 ceiling counts tasks, not numbers — stated explicitly in the prompt so the agent does not renumber to "fix" a high number.
  • No orphans accumulate, because the report pass may never delete and there is no per-task prune path in the sync (only whole-spec deleteSpec). Every ID that ever existed still names live work.
  • No code change is needed: TASK_LINE_RE is order-agnostic and scans the whole file, and syncTasksFromMarkdown needs no guard because the dangerous path is never entered.

3 · Decision story

Business decisions

D1 · When does a risk row deserve a task of its own?
chosen
Only when it names behavior no Sequencing step describes
Objective and mechanically applicable; the agent never has to judge importance.
Leave it to the agent's judgement
Marginally better at catching genuine gaps, but the same spec yields a different task count on every run.
Why: tested against the real 9-row risk table in audit-q3-ux-error-feedback/plan-report.html. The rule correctly eliminates rows answered by existing behavior ("the panel is already single-toast", "already a no-op") and rows that cite a step explicitly ("Sequencing step 8 is a grep sweep"), leaving exactly one candidate. The list does not inflate, and the spec's own Open Questions demanded a crisp test the agent could apply.

Technical decisions

D2 · How should the tasks pass locate sections inside the report?
chosen
Semantically — "find the table listing risks and the plan's answers"
Survives template evolution; costs some precision in which table is picked.
Anchor to heading text or card number
Exact today, but fails silently the day a heading is renamed — no error, just an empty report pass.
Why: the plan-report template and a future implement-report will evolve independently. A silent no-op is the worst failure mode available here: the run would report success while doing none of the cross-checking the spec exists for.
D3 · Read the report whole, or extract targeted sections? follows from D2
chosen
Read the whole file
Spends context on markup; at 418–472 lines that is affordable.
Grep for anchors and read only those ranges
Cheaper, but presupposes the anchors D2 just rejected.
Why: decided silently — semantic location requires the full content, so D2 forecloses the alternative. Measured on the three reports currently on main: 418, 428 and 472 lines.
D4 · What happens when a section cannot be located? follows from D2
chosen
Treat it as absent; finish the run normally
Cannot distinguish "template changed" from "nothing to find" — accepted.
Fail or warn per missing section
Surfaces template drift, at the cost of noisy runs on every report-less spec.
Why: decided silently — C4 already requires a clean exit when the whole report is missing, and the two situations are indistinguishable in effect. Both mean "no gaps of this kind to close".
D5 · Does rewording a task's description rebind its ID?
chosen
No — a clarified description is the same work
None material.
Treat a revised description as new work
Would renumber on every clarification, defeating the entire ID-stability rule.
Why: decided silently — the reconcile rules explicitly permit revision, so revision must not be a renumbering trigger or the two rules contradict each other.
D6 · Where does the ID↔work binding come from on regeneration?
chosen
tasks.md alone
One file, always present, carries the whole mapping.
tasks.md for the binding plus tasks.json for each task's status
Lets the flow treat pending IDs as safe to move — useful only if the flow moves IDs at all.
Why: revised after D8. First decided the other way, on the reading that the risk is "not uniform" — renumbering a pending task costs nothing while renumbering a completed one is corruption, so status had to be consulted. That framing quietly assumed renumbering happens at all. Once D8 removed it entirely, status became irrelevant to the binding, and the second file with it. Restores the spec's original recommendation.
D7 · How is S6 verified?
chosen
A new test under test/
Adds a file to a template-only spec; cheap and locks the property in.
Verify by inspection
Zero cost now, no protection against a future change to the parser.
Why: decided silently — the project already runs node --test test/ with seven test files, and S6 states a precise, cheap property. This verifies existing behavior; it is not the sync hardening the spec places out of scope.
D8 · Is an existing task ever renumbered? supersedes D6
chosen
Never — an ID keeps its work for the life of the spec
Numbers drift above the task count over time; the list is not ascending.
Renumber where it is provably safe
Tidier numbering, at the cost of status data, a code-side guard, and a correctness argument per task.
Why: this is the whole defence. specManager.js:441-443 overwrites a matched task's title in place while preserving its status, so a shifted ID transplants a completion flag onto work never done — but that path is reachable only by renumbering. Not renumbering makes the corruption unreachable rather than merely unlikely: no guard is needed in syncTasksFromMarkdown, no existing flow is touched, and the spec stays a template-only change. The cheaper option turned out to be the stronger one.

6 · Convergence log

  1. Iteration 1 — coverage audit: C5, C6 and C7 had no owning plan section. Added a "Preserved by construction" subsection to Architecture tying each to the mechanism that preserves it, and made the claude-code-only scope explicit in Files.
  2. Iteration 2 — reality check on Footprint: discovered parseFootprintMarkdown filters by basename, so two of the three sample-project doc files would be silently stripped. Recorded as a known limitation in Architecture rather than worked around, since fixing the filter is out of scope.
  3. Iteration 3 — task-derivability: Sequencing had four consecutive steps all editing the same prompt section, which would have produced four near-identical tasks. Consolidated to eight steps, each independently committable.
  4. Iteration 4 — format and coverage audit: five sections in order, every G1–G4 / C1–C7 / S1–S6 ID owned by a named section, zero TBDs, every referenced path and symbol verified in code or marked New, tasks derivable from Sequencing alone. Pass — converged.
  5. Iteration 5 — post-review, prompted by the question "why renumber at all?". The plan had drifted away from its own spec: spec.md Goal 4 already said an ID stays bound to its work without exception, and D6 reintroduced renumbering as a thing the flow does by reasoning about which IDs were "safe to move". That nuance was the source of everything downstream — reading tasks.json, classifying tasks by status, and an emerging argument for a code-side guard in syncTasksFromMarkdown. Added D8 (never renumber), reverted D6 to tasks.md alone, dropped the guard entirely, and added the ceiling-counts-tasks-not-numbers rule that the change makes necessary. Net effect: one fewer file read, no code change, corruption path unreachable rather than guarded.
