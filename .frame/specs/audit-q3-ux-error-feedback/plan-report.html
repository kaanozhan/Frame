1 · What & why

The Q3 audit's dominant renderer theme is silent failure: when something goes wrong, the user usually never finds out. Clicking "New Frame" when the backend fails does nothing — no toast, no message (four separate call-sites drop the failure). A failed tasks.json write looks identical to success because the TASK_UPDATED handler's failure branch is empty. Both confirm modals fire the destructive action on Enter even when Cancel is focused — a user who Tabs to Cancel and presses Enter still deletes their task. And if the workspace never loads at boot, the splash silently hides after 10 seconds and drops the user into a blank app. Underneath it all: there is no shared notification layer (four divergent private showToast copies, one of which interpolates its message into innerHTML unescaped) and twenty duplicate escapeHtml definitions.

The plan establishes one error/feedback discipline: a single body-mounted notify.error/success/info module (behavior lifted from the best existing copy, message set via textContent so it is HTML-safe by construction), a single escapeHtml helper, an explicit rule that every user-triggered create/write failure routes to notify.error, focus-respecting Enter in both confirm modals with Cancel focused first, and a "couldn't load workspace" state with Retry in the boot loader. The vocabulary split (code says "lane", UI says "Frame"/"Home") is documented as a deliberate convention rather than half-renamed, and the parked hidden project-action buttons are removed while their palette commands are kept and properly rewired. This is hardening, not redesign — no IPC channels change, no visuals are restyled beyond what one shared component requires.

3 · Decision story

Business decisions

D1 · What does Enter do in the confirm modals?
chosen
Respect the focused button
Standard platform behavior; needs focus management (initial focus on Cancel) so a blind Enter stays safe.
Enter always cancels
Safest possible and trivial to implement, but non-standard — users expecting Enter-to-confirm must click or Tab.
Why: Focus-respecting Enter is what every native dialog does; combined with Cancel-first initial focus and a safe-cancel fallback for ambiguous focus, the destructive path always requires an explicit choice.
D2 · Naming: finish the lane→frame rename, or document the split?
chosen
Document the split
"Lane" stays in code/DOM ids, "Frame"/"Home" in UI — zero regression risk, but the two vocabularies persist.
Finish the rename in code
One vocabulary everywhere, but a large risky diff that would dominate this hardening spec.
Why: PROJECT_NOTES records a deliberate 2026-06-11 decision to keep "lane" internal-only, and the spec's constraint says respect it. The rule becomes explicit via a naming-convention header comment in laneBoard.js; the PROJECT_NOTES entry lands at meta-reconciliation (spec workers never touch meta files).
D3 · What happens to the parked #parked-project-actions buttons?
chosen
Remove the DOM, keep the palette commands
No new UI surface; both capabilities live on in the command palette only.
Give them a real visible home
Restores discoverability, but that is new UI design — beyond hardening scope. (Removing the commands too was also rejected: it loses a working capability.)
Why: Evidence showed ai.startSession currently proxies a click on the hidden disabled button — a no-op. The fix extracts the real handler (index.js:255) into a named function the command calls directly; project.initializeFrame already calls state directly and needs nothing.

Technical decisions

D4 · Where do the shared helpers live?
chosen
notify.js + htmlUtils.js in src/renderer/
Two focused CommonJS modules; matches the flat one-concern-per-file renderer layout.
Single uiUtils.js grab-bag · or escapeHtml in src/shared/
One require per file, but couples unrelated concerns; shared/ holds only cross-process constants and nothing in main needs escapeHtml.
Why: The renderer directory is already flat with one concern per file (state.js, platform.js, appLoader.js…); DOM-adjacent helpers stay renderer-side and src/shared/ keeps its cross-process-constants meaning.
D5 · How is the boot-failure state triggered?
chosen
Renderer-only timeout state
No IPC changes; can't distinguish slow vs dead main, so the message says "couldn't load" with Retry.
Add a main-side error signal
Precise diagnostics, but a new IPC channel and main-process changes — beyond this renderer-focused spec.
Why: Main has no failure path to report — it always replies WORKSPACE_DATA (workspace.js:203) — so a main-side signal would mean inventing new failure detection in main. The timeout is the honest signal the renderer actually has, and the keep-channels-intact constraint applies.
D6 · Toast behavior baseline silent
chosen
tasksPanel variant, verbatim
Body-mounted, icon + message, single toast, 4000/2000 ms — behavior users already see.
Design a new toast behavior
Any improvement here is redesign — explicitly out of scope, and a 5th variant is exactly what the constraint forbids.
Why: The spec's consolidate-don't-fork constraint names this baseline (body-mounted, 4000 ms errors / 2000 ms otherwise per tasksPanel.js:412). The only behavioral change is the security fix: textContent instead of innerHTML interpolation.
D7 · Terminal-create call-sites: null-check only, or try/catch too? silent
chosen
try/catch + falsy-check, distinct messages
Slightly more code per site; both failure routes (cap and backend) become visible with accurate messages.
Falsy-check only (what the spec implies)
Matches the spec's literal reading, but leaves backend failures as unhandled promise rejections — still silent.
Why: Evidence pass found createTerminal rejects on backend failure and returns null only at the per-project cap (terminalManager.js:283-302). A null-check alone would fix half the bug; the success criterion is "no silent failure at all four sites".
D8 · Consolidation scope: the 20 copies, not the listed 15 silent
chosen
Migrate every definition found today
Five files beyond the spec's list (sampleBanner, terminalGrid, laneBoard, terminalTabBar, agentDispatch) join the migration.
Only the spec's enumerated files
Smaller diff, but the success criterion "exactly one escapeHtml" would fail on day one.
Why: The spec says "15+" and the criterion is exactly-one-helper; the evidence pass is the source of truth for what exists today.

4 · Risks & edges

Edge / riskPlan's answer
Backend create-failure is a promise rejection, not a falsy returnAll four sites get try/catch as well as the falsy-check (D7); cap and backend failures produce distinct notify.error messages.
Repeated failures spam toastsThe tasksPanel baseline is single-toast-at-a-time — a new call replaces the visible toast, never stacks.
Modal focus is lost (e.g. user clicked the backdrop) and Enter arrivesEnter falls back to cancel() when document.activeElement is not one of the modal's buttons — ambiguity resolves to the safe action.

WORKSPACE_DATA arrives after the failure state is shown (slow main, not dead)The loader's existing first-data listener still fires and hides the loader — the failure state self-resolves; Retry also re-arms the failsafe.
DOM-based vs string-replace escapeHtml behavior driftBoth approaches produce identical output for the five escaped characters; the shared helper is string-replace so it has no DOM dependency and one testable behavior.

Naming documentation vs the workers-never-touch-meta-files ruleThe convention lands as a header comment in laneBoard.js (in scope); the PROJECT_NOTES.md decision entry is appended at meta-reconciliation, outside the worker's footprint.

Deliberately deferred: the full lane→frame rename (documented split instead, per D2); a visible home for start-AI/init-frame (palette-only, per D3); a main-side workspace-load error signal (per D5); any toast/modal visual redesign (out of scope).
