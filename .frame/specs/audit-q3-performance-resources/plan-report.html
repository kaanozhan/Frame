1 · What & why

Frame's main process is the IPC server for every window, yet it does most of its work synchronously and rebuilds full datasets on every change. A Q3 audit verified the damage: 206 synchronous fs/child_process calls in src/main/ (including a spawnSync that can freeze the event loop for 30 seconds during project init), ~29 full tasks.json parses triggered by a single file change, unthrottled per-chunk PTY streaming to up to 9 terminals, four polling loops that never pause when the app is hidden, unbounded prompt logs, and PTYs orphaned by a window reload. The plan converts hot paths to async, makes task/spec pushes parse-once + skip-unchanged, coalesces PTY output with backpressure, gates every poll on window visibility, bounds logs and saved sessions, tears PTYs down on reload, and vendors D3 with a throttled force simulation.

This shape — fix the sources of waste rather than redesign the transport — was chosen because the datasets are small (237 tasks) while the redundancy is enormous. Keeping the existing IPC channels and payload shapes untouched preserves the spec's hard constraints (no contract changes, byte-identical behavior while visible) and keeps every step independently shippable.

3 · Decision story

Business decisions

D1 · What should a window reload do with running terminals?
chosen
Destroy and recreate
Simple and leak-free, but a reload interrupts long-running agent commands.
Persist and re-attach
Agents survive reload, but needs a re-attach handshake + scrollback replay protocol.
Why: matches the existing window-close semantics exactly; reload is rare, and a re-attach protocol is too much new surface for a perf-only spec.
D2 · When the window is hidden, do polls pause or merely slow down? silent
chosen
Pause fully, refresh immediately on show
Zero hidden-window cost; data can be stale only while nobody is looking.
Slow to a background cadence
Keeps data warm while hidden, but retains battery/CPU cost for no visible benefit.
Why: the show-refresh guarantees the user never observes staler data than today, which satisfies the keep-behavior-identical constraint while maximizing the saving.

Technical decisions

D3 · How do task/spec updates become incremental on the wire?
chosen
Parse-once + skip-unchanged at the source
Zero renderer changes and zero contract risk; payloads stay full-shape.
New TASKS_DIFF / SPEC_DIFF channels
Smallest wire traffic and scales furthest, but touches every consumer and doubles the IPC surface.
Why: at 237 tasks the wire size is not the cost — the ~29 redundant parses and up-to-28 status rewrites are. Fixing the source removes ~95% of the waste with none of the contract risk.
D4 · How do we measure block time, IPC cost, and startup reproducibly?
chosen
Lightweight in-app instrumentation (perfMonitor)
Reproducible on every run and cheap to keep, but less deep than full tracing.
Electron tracing + synthetic 9-PTY harness
Deepest visibility, but significant build/maintenance cost that outlives this spec.
Why: an event-loop-lag sampler plus startup marks and op timers is sufficient to prove every success criterion with before/after numbers.
D5 · Do heavy scans move to a worker, or just become async? silent
chosen
Async only — no worker_threads
Minimal new architecture; relies on the existing child-process parser.
Move scans to worker threads
Isolates CPU work fully, but adds a new runtime concept the codebase doesn't have.
Why: the one CPU-heavy scan already runs in a child process (spawnSync → the fix is async spawn); everything else is I/O-bound and fs.promises suffices.
D6 · What are the concrete budgets? silent follows from D4
chosen
Fixed defaults, measured by perfMonitor
TTI ≤ 1.5s · block ≤ 50ms/op · PTY flush 16ms · log cap 5MB+.1 · D3 ≤ 1500 nodes/~300 ticks · 20 MRU sessions.
Derive budgets from a benchmarking phase first
More principled numbers, but delays all implementation behind a measurement project.
Why: these defaults are conventional (16ms ≈ one frame, 50ms ≈ perceptibility threshold) and step 1 records baselines so the final measurement pass can prove them or adjust with data.
D7 · How is D3 vendored? silent
chosen
npm dependency + node_modules script tag
Matches the existing xterm.css pattern in index.html; no bundler work.
Check a minified copy into the repo
No dependency added, but an unversioned vendored blob to maintain by hand.
Why: index.html:6 already loads node_modules/xterm/css/xterm.css directly — the same mechanism works for node_modules/d3/dist/d3.min.js with zero build changes.
D8 · How are localStorage terminal sessions pruned? silent
chosen
Cap at 20 MRU + wire clearProjectSession to project removal
Bounded by count; the dead code finally gets a caller.
Age-based expiry
Handles abandonment gracefully, but needs timestamps the records don't carry today.
Why: records are tiny (viewMode/gridLayout/names), so a simple MRU count cap on saveProjectSession is sufficient and behavior-neutral.

4 · Risks & edges

Edge / riskPlan's answer
Making init async lets two IPC calls run project init concurrentlyThe init path returns a shared in-flight promise; a second call awaits the first instead of racing it.
did-start-navigation also fires on the first page loaddestroyAll() is idempotent — on first load there are no PTYs, so the hook is a no-op.
PTY pause() could stick if a flush never runsresume() is called unconditionally by the 16ms flush timer; a pause cannot outlive one flush interval.
Payload-hash skip could suppress a real update on hash collisionThe "hash" is an equality check on the serialized payload (cheap at this size) — no lossy hashing, no false skips.
Self-write guard window could swallow a genuinely external write that lands within the stamp windowGuard windows stay short (matching tasksManager's existing SELF_WRITE_GUARD_MS), and the payload-hash gate means a swallowed event would be corrected by the next real change; the debounce is retained as a safety net.
Coalescing changes terminal output pacing16ms is one display frame — xterm renders per frame anyway, so perceived fidelity is unchanged; fidelity of content is byte-exact.
Node cap changes the structure map on giant reposOnly repos over 1500 modules are affected; the cap keeps the highest-degree nodes and logs a visible notice — a degradation bound, not a redesign.
Prompt-log rotation discards history beyond the capBy design: 5MB + one .log.1 generation keeps ~10MB of recent history; getHistory reads become bounded by the same cap.
Async Keychain read overlapping a quitPoll intervals are cleared on quit as today; the TTL cache makes the last result reusable without re-exec.

Deliberately deferred: diff-message IPC channels (revisit only if datasets grow ~10×, would land as versioned channels alongside the full-payload ones); persist-and-re-attach PTYs across reload (would land as its own spec with a re-attach protocol); web/browser mode (out of scope by spec).
