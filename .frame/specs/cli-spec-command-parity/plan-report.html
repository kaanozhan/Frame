1 · What & why

Frame's spec commands only get their real, current-generation flow when the UI button dispatches them: the app interpolates the latest template, stages it plus its assets, and computes launch flags. When a user instead asks an already-running CLI session to "plan this spec", none of that fires — the agent falls back to project docs that describe an obsolete generation of the flow (four plan sections, no ## Footprint, no implement modes, no reports), and produces plans that silently break orchestration's collision detection. Most users drive Frame from the CLI, so the degraded path is the common path.

The plan makes the CLI path self-served with two mechanisms: (1) staging — on every project open, Frame copies the current four command templates and their two assets into .frame/runtime/commands/<tool>/, write-if-changed (the v2 implement slice already does exactly this for one command; this generalizes it); and (2) managed docs — the spec section of REFERENCE.md and AGENTS.md becomes a version-stamped, marker-delimited block Frame can upgrade in place, whose new text teaches the agent the self-serve protocol: resolve the spec deterministically, read the staged template (override first), interpolate, follow it exactly, never improvise from memory. This shape wins over slash-command shims because the observed failure is conversational entry — which only docs can catch — and because Frame has an explicit stance of never writing into the user's .claude/.

3 · Decision story

Business decisions

D1 · How does a CLI session discover and run the spec commands?
chosen
Docs-taught protocol only
Depends on the agent reading the CLAUDE.md → REFERENCE.md chain.
Also generate .claude/commands/spec.*.md shims
/spec.plan works verbatim, but tool-specific, a second staged surface, and writes into the user's .claude/.
Why: The observed failure is conversational entry, which only docs catch; shims would be a shortcut layered on top. One maintenance surface, tool-agnostic, and honors Frame's explicit "never write the user's .claude/" stance (specManager.js:434).
D2 · How does the agent pick the target spec from conversation?
chosen
Deterministic rule in the docs
A few lines of rule text in the managed block.
Agent judgment
Flexible, but two sessions can act differently on the same project state.
Why: Exactly one spec in the command's actionable phase → take silently; zero or several → ask with candidates; an explicitly named spec always wins. Parity means the choice itself must be predictable.

Technical decisions

D3 · Which files does the legacy migration cover?
chosen
REFERENCE.md + AGENTS.md
Touches two user-owned files (each under the same confidence gate).
REFERENCE.md only
Narrower, but leaves the always-on misleading section in old AGENTS.md files.
Why: Older Frame generations shipped the full obsolete section directly into AGENTS.md (frameTemplates.js:22-26 docstring) — and AGENTS.md loads every session, making it the primary drift source.
D4 · Should this spec's work ship with tests?
chosen
Pure logic and data transforms only
IPC/watcher wiring stays untested, per convention.
Everything testable / none this time
Stubbed integration adds maintenance; zero tests would skip exactly the kind of pure logic this project does test.
Why: The project's recorded convention targets pure modules and skips Electron-coupled wrappers; the block engine and staging-plan resolution are designed pure so the convention reaches them.
D5 · Staging refresh: project open only, or also a manual bin path? silent
chosen
Project open (WATCH_SPECS) + init/enable only
A project untouched by a newer Frame stays on its last-staged generation until next open.
Also a manual node .frame/bin/ refresh
Cannot work: a bin script can't read Frame's packaged templates (app.asar) with the app closed — nothing newer to copy from.
Why: Technically impossible to do better from outside the app; the protocol keeps implement-launch.js's existing answer — "open this project in Frame once".
D6 · Autonomous handoff wording resolved by shipped code
chosen
Keep v2's dual handoff: Frame button + exact launch line
None — already shipped and field-correct.
Frame button only
Strands terminal-only users.
Why: The spec's open question was answered by the v2 implement template (spec.implement.md:61-72), which already prints both. Recorded as drift-resolved; the managed block repeats the same handoff.
D7 · Version gate: integer stamp vs content compare silent
chosen
Monotonic integer in the begin marker
A forgotten bump delays an upgrade one release.
Content hash / byte compare
Self-maintaining, but clobbers any user tweak inside the block on every open.
Why: The spec mandates a version stamp; the integer preserves in-block user tweaks between Frame releases and makes "upgraded exactly once" (S4) trivially provable.
D8–D11 · Placement, paths, tools, confidence silent
chosen
Pure engine in src/shared/, staging in its own main module; staged-path interpolation; existing tool dirs only; heading + shipped-body byte match
One more module than strictly necessary (commandStaging.js).
Staging inside specManager/frameProject; agent-side asset copying; heading-only legacy match
Require cycle at init; extra protocol steps; heading-only matching would clobber customized sections (this repo's own docs are the counterexample).
Why: D8 avoids frameProject ↔ specManager cycles; D9 points interpolation at files S1 guarantees; D10 keeps the mechanism tool-agnostic with claude-code as the only populated tool (Out of Scope); D11's body-match gate is what makes S5 ("heavily rewritten → left alone") mechanically true — both legacy section texts shipped as unparameterized constants, so byte comparison is sound.

4 · Risks & edges

Edge / riskPlan's answer

Templates staged into non-Frame or spec-disabled projectsSame boundary as the shipped v2 slice (WATCH_SPECS trigger); docs upgrade additionally requires the file to already exist — nothing is ever created.
Stale staged templates in a project the app hasn't opened since a Frame updateAccepted (D5) — a bin refresh is impossible without the app; next open refreshes. The docs say so honestly.
Marker corruption (begin without end, malformed stamp)Engine treats unparseable markers as "cannot locate confidently" → no rewrite; covered by tests.

Deliberately deferred: codex/gemini template content (Out of Scope; staging mechanism ships tool-agnostic), slash shims (would be a future spec if wanted), in-session privilege escalation (Out of Scope).
