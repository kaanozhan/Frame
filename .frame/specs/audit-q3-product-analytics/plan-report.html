1 · What & why

Frame's telemetry today emits exactly one event — app_started — so the founder has zero signal on which features are used, whether users ever activate (init a project, create a spec, start an agent), or what fails in the wild. Worse, the opt-out is not fully durable: if user-settings.json is corrupted beyond its .bak's reach, the settings cache resets to {} and isEnabled() silently flips an opted-out user back on — a direct contradiction of PRIVACY.md.

The plan extends the existing Aptabase integration with a small, deliberate registry of ten content-free events (activation, feature usage, coarse error counts), all validated at runtime against a declarative allowlist so nothing outside a fixed enum can ever leave the machine. The opt-out bug is fixed first, as its own shippable step: an unrecoverable settings load now fails closed (telemetry off for the session) instead of failing open. PRIVACY.md is updated in the same change set so disclosure never lags shipping. This shape — extend, don't replace; allowlist, don't trust call sites — was chosen over adding a heavier vendor (PostHog) or relying on review-time convention, because the spec's constraints (privacy commitments, low cardinality, Aptabase-first) make those alternatives strictly worse.

Architectural notes
  • Single choke point: every event — main-process or renderer — passes through track(); the renderer's TELEMETRY_TRACK IPC channel lands on the same validation, so no call site can bypass the allowlist.
  • Registry is pure and testable: src/main/telemetryEvents.js (New) has no Electron imports, so node --test can assert the registry contains only enum-typed props — the mechanical "no PII" guarantee.
  • Cardinality control: normalizeTool() collapses user-defined tool ids to custom (and claude-code → claude); plugin identity is never sent, only enabled|disabled.
  • Ten events total: app_started (existing), project_initialized, spec_created, spec_phase_advanced, agent_run_started, orchestrator_opened, orchestration_run_started, plugin_toggled, ai_tool_selected, error_occurred.

Architectural notes
  • The bug being closed: today cache = data || {} (userSettings.js:33) plus value !== false (telemetry.js:70) means unrecoverable corruption silently re-opts a user in. The unrecoverable branch above is the fix.
  • ENOENT ≠ corruption: readJsonWithRecovery already distinguishes them (error: null vs set), so fresh installs keep the spec-mandated default-on opt-out model untouched.
  • Pure decision function: effectiveEnabled({value, loadFailed}) lives in telemetryEvents.js, making all four states unit-testable — the regression test success criterion S2 demands.

3 · Decision story

Business decisions

D1 · How is activation measured in the first cut?
chosen
Plain events only
Simplest set, no local state; activation read as Aptabase unique-users-per-event, so funnel precision is approximate.
Explicit first_* milestone events
Exact activation counts, but more events plus persistent client-side "first done" flags to maintain.
Why: Unique-user counts on project_initialized / spec_created / agent_run_started already answer "do users activate?"; the extra precision of milestone events isn't worth the added event sprawl and local state.
D2 · What happens when user-settings.json is unreadable and .bak can't recover it?
chosen
Silent fail-closed
Zero nag; a user who never opted out loses at most one session of data.
Fail-closed + re-show consent banner
More transparent, but nags users who never opted out and adds banner-state logic for a rare corruption case.
Why: Private-by-default with no UI cost; ENOENT (fresh install) is distinguished from corruption, so the default-on model the spec requires is preserved exactly where it should be.

Technical decisions

D3 · How is "no PII / no content" enforced against future contributors?
chosen
Runtime event registry (allowlist)
Small runtime cost and a registry to maintain — but enforcement is mechanical, not social.
Review checklist only
Zero code, but nothing mechanically stops a content-bearing property from shipping.
Why: track() drops unregistered events/props/values silently, and a unit test asserts the registry is enum-only — a contributor physically cannot ship free-form strings past it.
D4 · How do renderer-originated signals reach telemetry?
chosen
TELEMETRY_TRACK IPC channel
One new channel to wire, but every signal is captured at its true source.
Main-process-only proxies
No new channel, but "orchestrator opened" and precise agent-launch semantics are lost or approximated.
Why: The main process validates against the same registry before anything reaches Aptabase, so the renderer cannot bypass the allowlist — true-source capture with no trust extended.
D5 · Stay on Aptabase or evaluate PostHog? silent
chosen
Stay on Aptabase
Coarser funnel views, but zero migration and anonymous by design.
Switch to PostHog
Rich funnels/retention, but a heavier SDK and identity features the spec rules out of scope.
Why: The spec's constraint prefers extending Aptabase absent a concrete reason; every founder question is answerable from event counts + unique-user counts, which Aptabase provides.
D6 · API shape for sending events silent
chosen
track(name, props) in telemetry.js
One gated helper; trackAppStarted becomes a thin wrapper.
Per-event functions (trackSpecCreated, …)
More explicit signatures, but ten near-identical wrappers and a wider surface to keep gated.
Why: The spec's constraint literally suggests "a track(name, props) helper gated by isEnabled()" — matching the existing module's shape.
D7 · What goes in the tool / plugin props? follows from D3
chosen
Fixed enums; normalize everything else
Custom tools all read as "custom" on the dashboard — identity of user-defined tools is invisible by design.
Send raw tool/plugin ids
Full visibility, but user-defined tool names and marketplace plugin ids are user content — unbounded cardinality and a PII risk.
Why: normalizeTool() maps to claude|codex|gemini|custom (and claude-code→claude); plugin_toggled carries only enabled|disabled. Low cardinality is a spec constraint, not a preference.
D8 · Error taxonomy silent
chosen
One error_occurred event, fixed category enum
Nine categories derived from failure paths the code already handles; new categories require a registry edit.
Per-error events with details
Richer debugging, but messages/stacks/paths are exactly what the spec forbids sending.
Why: Counts and categories answer "what are the most common in-the-wild errors?" without any content; the enum is seeded from error sites the evidence pass located in code.
D9 · How do telemetry tests run under node --test? silent
chosen
Extract pure telemetryEvents.js module
One new file; registry, validation, normalization, and the enabled decision all testable without Electron.
Test telemetry.js directly with mocks
No new file, but telemetry.js requires @aptabase/electron and electron itself — heavy mocking in a runner with no built-in module mocks.
Why: Matches how fsSafe/logger are tested today; the S2 regression test and the S1 enum-only assertion both land in plain node --test.
