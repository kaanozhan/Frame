Speaker notes — 5/16
The dev tools of the last 10–15 years were built for a pre-AI world; they're all trying to bolt AI on. I've built software professionally for over a decade, so Jira and these tools were mine too. On task management we used to record a task, assign it, and wait days for something an agent now does in ten minutes — with real doubts about whether the process was even followed honestly. And documentation is left to whoever implements it. With AI, context matters more than ever, and the AI itself should produce it. We believe AI belongs at the center and the tools rebuilt around it. The market seems to agree — Atlassian's stock took a hit in the AI era.
Speaker notes — 6/16
This isn't a niche. A whole category has appeared for running agents in parallel — Conductor, Crystal, Vibe Kanban, Cursor's background agents, Devin — drawing real attention and funding, which validates the need. But most are agent runners: they launch several agents, often in git worktrees. Frame sits a layer above — planning-first, with specs, durable context, orchestration, and a workspace spanning Claude, Codex, and Gemini. The difference in one line: they parallelize prompts; we parallelize plans.
Speaker notes — 7/16
The core problem with agents is that they're stateless. Every new session starts from zero — no memory of your architecture, your decisions, or the reasoning behind the code. The popular answer, bigger context windows, doesn't fix it: you can't paste a whole project into every prompt, and raw code isn't the reasoning that produced it. What you need is structured context that lives outside the conversation — captured at the right moments and fed back automatically. Everything in Frame is built around that.
Speaker notes — 8/16
Spec-driven development means significant work goes through a spec before any code: what you're building, how, broken into tasks, and what actually shipped — all plain markdown in your repo. It fits agents naturally: a plan beats a vague prompt because the spec forces the thinking up front, the way good engineering always worked. The agent then follows ordered tasks instead of improvising, and records the outcome while it's fresh.
Speaker notes — 9/16
This is the part that solves the context problem. The spec workflow doesn't just guide the build — it generates structured context as a side effect. The plan, tasks, and especially the outcome become the project's durable memory, written by the AI at the moments that matter. Because it's plain files, any tool reads it, git versions it, teammates grep it. Frame ships this as a first-class workflow with tasks importing automatically — optional, never forced.
Speaker notes — 10/16
Agents made each individual unit of work fast, so the bottleneck moved: now it's about doing many units at once. But run agents naively and they collide — editing the same files, undoing each other's work. This is where specs pay off again: a spec is already a self-contained unit that declares its footprint — the files it touches. That's exactly what you need to parallelize safely. You can't safely parallelize vague prompts; you can parallelize specs.
Speaker notes — 11/16
In Frame you hand a set of ready specs to a conductor agent. It reads each spec's footprint and runs only the non-overlapping ones in parallel — each agent isolated in its own git worktree, on its own branch, so no two touch the same files. That safety is enforced in code, not a prompt. And you stay in control: watch every agent on a live pipeline, drop into any of them, and nothing merges until you approve it. Your main branch is never touched automatically. Guardrailed, human-steered — not fire-and-forget.
Speaker notes — 12/16
One more benefit that matters for cost: our approach uses fewer tokens. Without structure, agents burn tokens rediscovering the project every session and reworking wrong turns from vague prompts. With Frame, structured context is read once and automatically; the intentIndex sends agents straight to the right files instead of scanning the whole codebase; and a plan up front avoids rework — so the tokens go to the actual work. Each spec also loads only its own footprint, not the entire project. Better plans don't just produce better code; they make each run cheaper.
Speaker notes — 13/16
The practical payoff: with Frame you're more organized, faster, and leaner. More organized — every project, spec, task, and agent live in one workspace on a standard structure, so nothing is scattered or lost. Faster — you plan once and run agents in parallel, and each agent picks up full context instantly instead of re-learning the project every session. More efficient — tokens and time go to the actual work, not rediscovery or rework. Same agents, but more output with far less overhead.
